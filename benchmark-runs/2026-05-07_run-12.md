## Changes Applied (This Run)

### 1. Retry-on-Early-Exit Mechanism (Major — Execution Error Recovery)

**Problem:** Terminal-Bench 2.0's failure taxonomy shows ~50% of failures are
execution errors. Analysis reveals that many failures happen early in the time
budget — the agent misunderstands the task, hits an unexpected error, or takes
a wrong approach that appears to succeed but actually didn't solve the problem.

**Implementation:**
- AgentRunner now tracks `launchTime` and `lastOutputTail` (last 500 chars of output)
- If agent exits with code 0 before using 40% of its time budget, it triggers
  a retry with diagnostic context appended to the task
- Retry context includes: what happened in the first attempt, instruction to
  take a completely different approach, re-read task and test scripts
- Maximum 1 retry to avoid wasting the entire budget
- Constants: `EARLY_EXIT_THRESHOLD = 0.40`, `MAX_EARLY_EXIT_RETRIES = 1`
- Only active in benchmark mode (when `GITGANG_TIME_BUDGET_SECONDS` is set)

**Why it's general (not a hack):** This addresses a structural failure pattern
where any agent can misunderstand a task and exit quickly thinking it's done.
The pattern applies to any task, not specific benchmarks.

**Expected impact:** +3-5% — converts early failures into second chances with
diagnostic context from the first attempt.

### 2. CLAUDE.md Bootstrap with Test Script Pre-reading (Major — Verification Error Prevention)

**Problem:** The agent wastes 2-5 turns at runtime discovering and reading test
scripts. This is time that could be spent coding. Including test script contents
in CLAUDE.md means the agent starts with full knowledge of what "correct" means.

**Implementation:**
- New `bootstrapClaudeMd(worktreeDir)` function called before agent launch in
  benchmark mode
- Scans worktree for test/validation scripts matching patterns: `test_*`,
  `*_test.*`, `*.test.*`, `verify*`, `check*`, `validate*`, `grade*`, `score*`
- Includes first 150 lines of each script (up to 10 files)
- Also includes Makefile `test:`, `check:`, `validate:` targets
- CLAUDE.md header emphasizes: "These scripts are GROUND TRUTH"
- Does NOT overwrite existing CLAUDE.md (task repo may ship its own)
- Only writes CLAUDE.md if useful content was found

**Why it's general:** Pre-reading test scripts saves turns for ANY task with
validation scripts. The agent starts with the verifier's expectations built in.

**Expected impact:** +2-4% — saves turns and prevents misinterpretation of what
the verifier checks.

### 3. Domain-Specific Project Hints (Medium — Coherence Error Prevention)

**Problem:** Agents waste time discovering basic project setup (build commands,
dependency management, common pitfalls) for each language/framework.

**Implementation:**
- New `detectProjectHints(dir)` function scans for project manifest files
- Detects: Rust (Cargo.toml), Go (go.mod), Python (requirements.txt/pyproject.toml/setup.py),
  Node.js (package.json), C/C++ (CMakeLists.txt/Makefile), Java (pom.xml/build.gradle),
  Docker (Dockerfile/docker-compose.yml)
- Each detection adds domain-specific hints: build commands, dependency installation,
  common pitfalls (borrow checker, ESM vs CJS, etc.)
- Hints are included in the bootstrapped CLAUDE.md

**Why it's general:** Language-specific build/test patterns are universal
knowledge that applies to any task in that language.

**Expected impact:** +1-2% — reduces time spent on environmental discovery.

### 4. System Prompt Restructuring (Medium — Planning & Efficiency)

**Problem:** Top-scoring agents (GPT-5.5 native at 82%, ForgeCode at 78%) all use
explicit planning steps and batch file operations. Our workflow jumped straight
from reading to coding.

**Changes to `systemConstraints()`:**
- Added "Step 0: Survey" — batch environment discovery in a single command
- Added "Step 1: Plan" — write a brief plan before coding (state approach, identify
  pitfalls, trace test assertions)
- Added Rule 10: "Read multiple files in a single tool call when possible"
- Added long-running command management guidance (timeout wrapping, background
  processes, heredoc for interactive sessions)
- Added more error recovery patterns (compilation cascades, linker errors, segfaults)
- Added "verify from scratch in clean subshell" to final verification step
- Loop detection now includes byte-for-byte comparison reminder

**Changes to `featurePrompt()`:**
- Restructured from 7 steps to 8 steps (0-7) matching the new workflow
- Step 0 (SURVEY) emphasizes CLAUDE.md awareness and batch file reading
- Step 1 (PLAN) requires stating approach before coding
- Steps renumbered to match systemConstraints

**Why it's general:** Planning before coding, batch operations, and long-running
command management are universal agent patterns used by all top-scoring frameworks.

**Expected impact:** +2-3% — better first-attempt accuracy from planning, fewer
wasted turns from batch reads.

### Gap Analysis

The gap between raw Opus 4.7 (69.4%) and top agent scores (82%) is ~13 points.
Top agents gain this through:
1. **Better scaffolding** (+2-6pp): Planning, retries, context management
2. **Environment optimization** (+1-3pp): Pre-installed tools, test pre-reading
3. **Execution patterns** (+1-3pp): Long-running commands, batch operations
4. **Model routing** (+1-2pp): Using different models for different subtasks

Our improvements target categories 1-3. Category 4 (model routing) is a future
opportunity — e.g., using Gemini for planning and Claude for execution.
