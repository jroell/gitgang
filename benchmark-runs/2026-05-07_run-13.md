## Research Basis

### Harness Engineering Literature Review (May 2026)

Key findings from recent harness engineering research:

1. **AHE (Agentic Harness Engineering) paper** — Iterations lift pass@1 on
   Terminal-Bench 2 from 69.7% to 77.0%. Crucially, ablations localize gains to
   **tools, middleware, and long-term memory** rather than system prompt prose.
   This means factual harness structure transfers while prose-level strategy does not.

2. **HumanLayer "Skill Issue" article** — "70% of agent performance lives outside
   the model." The four customization levers: system prompt, tools/MCPs, context,
   and sub-agents. Sub-agents as "context firewalls" is a powerful pattern.

3. **Addy Osmani on harness engineering** — The most important habit is treating
   agent mistakes as permanent signals. Autonomous coding loops that re-inject
   prompts with filesystem state from the previous iteration are key.

4. **Claude Code native harness** — Achieves 92.1% on terminal-bench 2.0 when
   properly configured, vs ~69.4% for raw Opus 4.7. The gap is almost entirely
   harness engineering.

### Implication for This Run

Rather than further polishing system prompt prose (diminishing returns per AHE
research), this run focuses on **structural improvements** that give the agent
more actionable context before it starts:

- Pre-reading source files (not just test files)
- Pre-flight test execution (baseline output)
- Preserving task-provided CLAUDE.md content
- Enhanced retry with git diff and test result context

## Changes Applied (This Run)

### 1. CLAUDE.md Augmentation Instead of Skip (Medium — Context Completeness)

**Problem:** Previously, if a task repo shipped its own CLAUDE.md, gitgang
skipped bootstrapping entirely. This meant the agent lost all pre-analyzed
test scripts, project hints, and ground rules when a task provided its own
CLAUDE.md — which is exactly when context is most valuable.

**Implementation:**
- `bootstrapClaudeMd()` now reads any existing CLAUDE.md and preserves it
- Appends gitgang context (test scripts, hints, ground rules) after a separator
- Ensures the agent gets BOTH task-specific instructions AND gitgang's analysis

**Why it's general:** Task repos may provide useful domain context in CLAUDE.md.
Augmenting rather than replacing ensures both sources of truth are available.

**Expected impact:** +1-2% — tasks with their own CLAUDE.md now get full context.

### 2. Source File Pre-Reading in CLAUDE.md (Medium — Turn Efficiency)

**Problem:** Agents waste 2-4 turns reading source files at runtime. For tasks
with a clear set of implementation files, pre-loading them into CLAUDE.md means
the agent starts with the full codebase context.

**Implementation:**
- New `findSourceFiles(dir)` function scans for implementation files
- Filters by common source extensions (.py, .js, .ts, .rs, .go, .c, etc.)
- Skips test files, lock files, minified files, and files >15KB
- Caps at 8 files, 100 lines each to avoid bloating context
- Skips node_modules, .git, vendor, target, build, dist directories

**Why it's general:** Pre-reading source files saves turns for ANY task.
The agent starts with knowledge of what exists and how the code is structured.

**Expected impact:** +1-3% — saves 2-4 turns and prevents misunderstanding
of existing code structure.

### 3. Pre-flight Test Execution (Major — Baseline Establishment)

**Problem:** The agent doesn't know what state tests are in before it starts
coding. It may waste time trying to fix pre-existing failures or may not
realize tests already pass for some components.

**Implementation:**
- New `runPreflightTests(dir, testFiles)` function runs tests BEFORE agent starts
- Tries common test commands first: `make test`, `npm test`, `cargo test`, etc.
- Also runs executable test scripts directly (shebang or exec bit)
- Captures both stdout and stderr with 15s timeout per command
- Includes output in CLAUDE.md under "Pre-flight Test Output" section
- Total output capped at 3000 chars to avoid context bloat

**Why it's general:** Knowing baseline test state is universally useful. The
agent can see which tests fail and what their error messages look like before
writing any code.

**Expected impact:** +2-4% — agent starts with concrete knowledge of what's
failing and what the expected output format looks like.

### 4. Enhanced Early-Exit Retry with Git Diff and Test Results (Medium — Retry Quality)

**Problem:** The previous retry mechanism only included the last 500 chars of
agent output. This is often insufficient to understand what went wrong — the
agent needs to see what files were changed and what tests say.

**Implementation:**
- Increased output tail from 500 to 2000 chars for richer context
- Retry now captures `git diff HEAD` to show what files the previous attempt changed
- Retry also runs test scripts to capture their current state
- Instructions changed from "start from scratch" to "build on partial progress"
- Added explicit instruction to check `git status && git log --oneline -5`

**Why it's general:** Filesystem-level context (diffs, test results) is far more
actionable than raw output text. This applies to any task where the first attempt
makes partial progress.

**Expected impact:** +1-2% — retry attempts are now much more informed and can
build on previous work.

### 5. Additional System Prompt Rules (Minor — Failure Prevention)

**Problem:** Certain failure patterns recur across tasks: giving up too early,
not reverse-engineering test assertions, rewriting code that partially works.

**New rules added:**
- Rule 11: "NEVER give up" — use every second of time budget
- Rule 12: Reverse-engineer test assertions (work backwards from assertion)
- Rule 13: Prefer extending existing code over rewriting from scratch
- Rule 14: Create parent directories with `mkdir -p` before writing files
- Ground rule in CLAUDE.md: "Commit EARLY" reminder

**Why it's general:** These address universal agent failure patterns, not
task-specific issues.

**Expected impact:** +0.5-1% — prevents common failure modes.

### Key Insight from AHE Research

The AHE paper shows that moving from 69.7% → 77.0% comes primarily from
tools, middleware, and long-term memory — NOT system prompt changes. This
validates our approach of investing in structural improvements (pre-flight
testing, source pre-reading, filesystem-aware retries) over prompt engineering.
