## Changes Applied (This Run)

### 1. "Analyze Before Coding" Step (Major — Apex2-inspired)

**Research basis:** Apex2 (64.5% on tbench 2.0) achieves SOTA through intelligent
information gathering and sophisticated synthesis rather than complex multi-agent
orchestration. Key insight: "SOTA models have the answers — focus on extracting
the insights from them." Their ablation studies show knowledge extraction and
alternative approach planning are the highest-impact techniques.

**Implementation:**
- Added "Step 2: Analyze before coding" to systemConstraints():
  - Think about domain knowledge that applies (crypto standards, ML conventions, etc.)
  - Identify likely pitfalls for the task type (off-by-one, encoding, precision, etc.)
  - Consider 2 approaches and pick the most reliable one
  - Read existing code/tests to understand expected interfaces
- Workflow changed from "Read → Code → Commit → Test → Fix → Commit"
  to "Read → Analyze → Code → Commit → Test → Fix → Commit"
- This step is explicitly marked as "short (seconds, not minutes)" to avoid
  over-planning, which would waste time budget

**Why this helps:** Many tbench failures come from jumping straight to coding
without understanding the problem domain. A 10-second analysis phase prevents
5+ minutes of wrong-direction iteration.

### 2. Read ALL Source Files Before Coding (Major)

**Research basis:** Top-scoring agents (Meta-Harness, Terminus-KIRA) read existing
test files and source code before writing solutions. This reveals expected interfaces,
output formats, and edge cases that the task description may not fully specify.

**Implementation:**
- Step 1 now says "Read ALL source files the task references or that already exist"
- Step 1 explicitly includes: "Available test/validation scripts" and expected output analysis
- Feature prompt step 1: "Read ALL existing source files and test files in the repo"
- CLAUDE.md bootstrap: "Read ALL existing source files and test files before writing any code"
- Loop detection now includes: "Read the test/validation script source code if available"

### 3. Read Test Script Source When Stuck (Medium)

**Research basis:** LangChain's harness engineering blog showed that agents often
misunderstand what the verifier checks. Reading the actual test script source
reveals the exact assertions and expected behavior.

**Implementation:**
- Added to Loop Detection: "Read the test/validation script source code if available to understand what it expects"
- Added to feature prompt: "read the test/validation script source if available"
- Added to CLAUDE.md: "If test/validation scripts exist, read their source code to understand what they check"

### 4. Max Turns Reduced: 200 → 150

**Research basis:** Most tbench tasks complete in under 100 turns. 200 turns allowed
the agent to spend excessive time on loops without fundamental approach changes.
150 turns is still generous for complex tasks while creating more pressure to
pivot approaches earlier.

**Implementation:**
- `runClaude()` default benchmark max_turns: 150 (was 200)
- Harbor agent GITGANG_MAX_TURNS env: "150" (was "200")

### 5. Pair Mode Model Update

**Implementation:**

### 6. Build Script Resilience

**Implementation:**

- Fixes build failures in Docker containers and sandboxed environments

### 7. Stronger Pre-Verification Emphasis

**Implementation:**
- Step 5 (verification) now says "A solution that passes verification is worth
  infinitely more than a clever one that doesn't"
- Added: "If you can run the grading/test script yourself, do it now and read
  EVERY line of output"
- Before finishing: added step "If the test output shows ANY failure, fix it
  before committing"

### 8. CLAUDE.md Bootstrap: Python Package Inventory

**Implementation:**
- Added "Installed Python Packages" section to bootstrap CLAUDE.md
- Lists `pip list --format=columns` output (first 50 packages)
- Agent knows what's pre-installed vs what needs `pip install`
- Saves time and avoids mid-task install failures

## Strategy Analysis

### Why These Changes Should Improve Scores

1. **Analyze before coding (+3-5%)**: The single highest-impact change. Prevents
   the #1 failure mode: coding before understanding the problem. Apex2 showed
   that knowledge extraction is the most important scaffold technique.

2. **Read all source files (+2-3%)**: Many tasks include test files, existing code,
   or reference implementations. Reading these before coding reveals exact
   requirements the task description may not fully specify.

3. **Read test scripts when stuck (+1-2%)**: Prevents wasted loops where the agent
   repeatedly fixes the wrong thing because it doesn't understand what the test
   actually checks.

4. **Max turns 150 (+0-1%)**: Mild improvement. Creates pressure to pivot earlier
   rather than burning 200 turns on the same failing approach.

5. **Build resilience (+reliability)**: Prevents agent installation failures in
   Docker containers, which was a root cause of many run-1 failures.
