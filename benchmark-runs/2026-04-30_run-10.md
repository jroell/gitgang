## Changes Applied (This Run)

### 1. Effort Level: high → xhigh for Benchmark Mode (Major)

**Research basis:** Opus 4.7 (released April 16, 2026) substantially improved xhigh
performance and added native self-verification. The timeout issues that caused
LangChain to downgrade to "high" were specific to Opus 4.6. Anthropic now
recommends xhigh as the default for coding/agentic use cases with Opus 4.7.
Opus 4.7's SWE-bench Verified score jumped from 80.8% to 87.6%, and Terminal-Bench
2.0 native score went from 65.4% to 69.4%. The xhigh effort level produces
measurably better scores on complex reasoning tasks without the timeout penalty
that existed in Opus 4.6.

**Implementation:**
- `runClaude()` now uses `--effort xhigh` unconditionally (was "high" for benchmark)
- This aligns benchmark mode with the model's recommended operating point

**Expected impact:** +3-5% — this is the single highest-impact change. Better
reasoning per turn means fewer wasted turns, more correct first attempts, and
better error recovery. Opus 4.7's native self-verification at xhigh means the
model proactively validates its own work.

### 2. Working Directory Awareness (Major — Execution Error Prevention)

**Research basis:** Terminal-Bench 2.0's ICLR 2026 paper categorizes failures into
Execution Errors (~50%), Coherence Errors (~25%), and Verification Errors (~25%).
The dominant failure mode — execution errors — includes "disobeying explicit
requirements" and agents writing files to wrong directories. Analysis of the
error traces shows agents frequently lose track of their working directory during
multi-step tasks, especially when tasks use absolute paths like `/app/`.

**Implementation:**
- Added Rule 7: "Track your working directory. Run `pwd` before and after major
  operations. If the task specifies an absolute path like `/app/`, work there."
- Step 1: Added "What directory should your output files be in? Check if paths
  are absolute (like /app/) or relative."
- Step 3: Added "After writing files, verify they exist: `ls -la /path/to/file
  && head -5 /path/to/file`"
- Step 4: Changed `git add -A` to `cd $(git rev-parse --show-toplevel) && git
  add -A` to ensure commits happen from the right place
- Step 6: Same `cd $(git rev-parse --show-toplevel)` pattern for fix commits
- Loop Detection: Added "Check your working directory: `pwd` — are files ending
  up in the right place?"
- Before Finishing: Added `ls -la /exact/path/to/file` verification
- Feature prompt: Added `pwd` to step 1, file verification to step 3, directory
  check reminder before final commit
- CLAUDE.md bootstrap: Added working directory tracking and file verification
  instructions

**Expected impact:** +2-4% — directly addresses the #1 failure category (50% of
all failures). Agents that know where they are don't write files to wrong paths.

### 3. Harbor: procps Pre-installation (Medium — Infrastructure)

**Research basis:** Terminal-Bench documentation notes that "Some Docker images
don't have procps pre-installed, which affects dozens of tasks and causes Agent
runtime crashes when the ps command doesn't exist, throwing an ENOENT error."
This was a known source of failures in run-1.

**Implementation:**
- Added `procps` to the apt-get install list in the harbor agent's install() step

**Expected impact:** +1-2% — eliminates a category of infrastructure-level failures.

### 4. Harbor: Java JDK Pre-installation (Small — Infrastructure)

**Research basis:** Terminal-bench 2.0 includes tasks requiring Java compilation
and execution. Tasks that need javac/java fail when these aren't available and
mid-task installation eats significant time budget.

**Implementation:**
- Added `default-jdk-headless` as a best-effort install (won't fail if unavailable)

**Expected impact:** +0.5-1% — eliminates Java toolchain as a failure mode.

### 5. Harbor: Additional Python Packages (Small)

**Implementation:**
- Added `scikit-learn`, `regex`, `chardet` to pre-installed packages
- These cover ML tasks, advanced text processing, and encoding detection

### 6. Improved Error Recovery Guidance (Small)

**Implementation:**
- Expanded error recovery section with specific patterns for import errors and
  version mismatches
- Added guidance to check what's actually installed before assuming a module works
- Separated quick-fix patterns for better readability

### 7. Stronger Test Output Reading (Small)

**Implementation:**
- Step 5: Added "If test output is long, DON'T skip reading it. Every line could
  reveal a failure pattern."
- This addresses the verification error category where agents glance at test
  output without reading it thoroughly

## Strategy Analysis

### Why These Changes Should Improve Scores

1. **xhigh effort (+3-5%)**: Most impactful. Opus 4.7's xhigh is substantially
   better than high for complex reasoning, and the timeout issues from Opus 4.6
   are resolved. Better reasoning = fewer wasted turns, more correct first
   attempts, better self-verification.

2. **Working directory awareness (+2-4%)**: Directly targets the #1 failure
   category. Simple but effective: making the agent explicitly track where it is
   prevents a large class of "files in wrong place" failures.

3. **procps installation (+1-2%)**: Infrastructure fix that eliminates a known
   crash category across multiple Docker-based tasks.

4. **Java JDK (+0.5-1%)**: Eliminates Java toolchain failures.

5. **Additional packages & error guidance (+0-0.5%)**: Minor but cumulative.
