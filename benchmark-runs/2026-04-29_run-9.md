## Changes Applied (This Run)

### 1. System Prompt / User Prompt Separation (Major — Architectural)

**Research basis:** Claude Code's `--append-system-prompt-file` flag allows injecting
additional system-level instructions that get stronger instruction adherence than content
passed as user messages. Previously, gitgang combined `systemConstraints()` and
`featurePrompt()` into a single string piped via stdin as a user message. Now:

- `systemConstraints()` → written to file → passed via `--append-system-prompt-file`
  (becomes part of the system prompt)
- `featurePrompt()` → piped via stdin as the user message

This separation means the workflow rules, error recovery patterns, and loop detection
guidance are treated as system-level instructions (which Claude follows more reliably)
while the task-specific content comes as the user request.

**Implementation:**
- `runClaude()` now writes two files: `claude-system-prompt.txt` and `claude-prompt.txt`
- Args include `--append-system-prompt-file <path>` for system constraints
- Only the user prompt is piped via stdin
- Added `--exclude-dynamic-system-prompt-sections` in benchmark mode for better
  cross-turn prompt cache reuse

### 2. "Read Test Script Source" Emphasis (Medium)

**Research basis:** LangChain's harness engineering blog and Apex2's ablation studies
both show that agents frequently misunderstand what the verifier checks. Reading the
test script source code reveals exact assertions and expected behavior that the task
description may not fully specify.

**Implementation:**
- Added Rule 6: "Read ALL existing source files, test files, and validation scripts
  before writing code. They reveal exact expected behavior."
- Step 1: Added "READ THEIR SOURCE CODE to understand what 'correct' means" for
  test/validation scripts
- Step 2: Added "What does the test/validation script actually check? Match THAT,
  not just your interpretation of the task."
- Step 5: Added "If a grading/test script exists, run it AND read its source code
  to understand what it checks."
- Step 5: Expanded test script glob to include `grade*` and `score*` patterns
- Step 6: Added "After each fix, re-read the task description to make sure you
  haven't drifted from requirements."

### 3. Analysis Step Time Cap (Small)

**Research basis:** Apex2 notes that knowledge extraction is high-impact but must be
fast. Previous prompt said "seconds, not minutes" — now explicitly says "<30 seconds"
and "Extract relevant domain knowledge from your training" to signal that this is about
recall, not exploration.

### 4. Harbor Agent: Go Toolchain Pre-installation (Medium)

**Research basis:** Terminal-bench 2.0 includes tasks requiring Go. We already
pre-install Rust but were missing Go. Some tasks that could otherwise pass fail
because `go` isn't available and mid-task installation eats time budget.

**Implementation:**
- New install step (2b) downloads and installs the latest Go release
- `$HOME/go/bin` added to PATH in all subsequent exec_as_agent calls

### 5. Harbor Agent: CLAUDE.md Task-First Restructuring (Medium)

**Research basis:** Claude Code reads CLAUDE.md from top to bottom. The task description
is the most important context — it should come first, not after environment details.
Previously, CLAUDE.md structure was: Instructions → Working Directory → Files → Tools →
Build Config → Task Description → Tests → Packages → System. Now it's:
Instructions → **Task Description** → **Test/Validation Scripts** → Working Directory →
Files → Tools → Build Config → Packages → System.

This means the agent has task context immediately when it starts processing, before any
environment details that are less critical for understanding what to do.

### 6. Harbor Agent: PATH Consistency

Updated all `exec_as_agent` PATH exports to include `$HOME/go/bin` alongside
`$HOME/.local/bin` and `$HOME/.cargo/bin` for consistent tool availability.

## Strategy Analysis

### Why These Changes Should Improve Scores

1. **System prompt separation (+2-4%)**: This is the most impactful change. System-level
   instructions have stronger adherence than user-message instructions in Claude Code.
   The workflow rules, loop detection, and error recovery guidance will be followed more
   consistently when they're in the system prompt.

2. **Test script source reading (+1-3%)**: Many tbench failures come from misinterpreting
   what the verifier checks. Explicitly reading test source code aligns the agent's
   understanding with the actual verification criteria.

3. **Task-first CLAUDE.md (+0.5-1%)**: Reduces the chance of Claude spending early turns
   on environmental exploration that's already answered in the file. Task context first
   means the agent starts reasoning about the problem immediately.

4. **Go toolchain (+0.5-1%)**: Eliminates a category of failures where Go-based tasks
   fail due to missing toolchain.

5. **Prompt cache reuse**: `--exclude-dynamic-system-prompt-sections` improves prompt
   caching across turns, reducing latency and cost per turn.

### General Patterns (Not Task-Specific)

All improvements are general-purpose:
- System prompt vs user prompt separation applies to any Claude Code invocation
- Reading test source code applies to any verification-based task
- Task-first context ordering applies to any bootstrapped environment
- Go toolchain is a general capability, not a task-specific dependency
- Prompt cache optimization is infrastructure-level
